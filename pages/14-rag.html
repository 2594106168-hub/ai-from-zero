📚 RAG & 向量数据库

LLM不懂你的公司文档。RAG在回答前先查文档——答案有据可查，不是编的。

1. 为什么需要RAG

LLM只有训练截止日前的知识。它不知道你公司昨天的销售数据，不知道你的产品手册，不知道你的内部文档。RAG在LLM收到问题后先去你的文档库检索相关内容→把文档和问题一起发给LLM→LLM基于文档回答。答案来自你的文档，非幻觉。

2. Embedding — 把文字变成数学向量

Embedding是把一段文字转成一串数字（向量）。语义相近的文字在向量空间中距离更近。"猫"和"猫咪"的向量几乎重叠，"猫"和"汽车"离很远。

Python 
from openai import OpenAI
c = OpenAI(api_key="sk-", base_url="https://api.deepseek.com/v1")
r = c.embeddings.create(model="text-embedding-3-small", input="你好世界")
print(len(r.data[0].embedding))  # 1536个数字 = 这句话的"语义指纹"

3. RAG完整流程

  1. 分块（Chunking）：把你的文档切成小段（每段300-1000字）
  2. 向量化：每段转成向量，存进向量数据库
  3. 检索：用户提问→把问题也转成向量→在库里找最相似的Top-K个文档片段
  4. 增强：把"文档片段+用户问题"拼成prompt，发给LLM
  5. 生成：LLM基于文档回答，给出引用来源

4. 向量数据库选型

数据库适合部署
Chroma学习/小项目pip install chromadb 直接本地跑
Pinecone生产环境托管服务，有免费额度
Qdrant高性能自托管或云服务
Supabase pgvector已用PostgreSQL直接挂扩展

5. 一句话总结

💡RAG = 搜索 + LLM。搜索负责"找什么"，LLM负责"怎么说"。先把相关内容找出来，再让AI基于真实数据组织语言回答。这样就杜绝了AI的"瞎编"。
