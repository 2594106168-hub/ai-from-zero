1.1 计算机怎么工作 — 4 个核心部件

🧠

CPU（处理器）

计算机的大脑。执行指令、做加减乘除。主频（GHz）越高越快，核心越多越能同时干多件事。Intel 和 AMD 两家垄断。

💾

RAM（内存）

CPU 的工作台。程序从硬盘加载到内存，CPU 直接在内存上读写。断电后数据消失。现在主流 16-32GB。

💿

硬盘（SSD/HDD）

永久存数据的地方。SSD 比 HDD 快 10-50 倍。你的代码、照片、游戏全在这。现在标配 512GB-1TB SSD。

🖥️

操作系统（OS）

管理硬件和软件的大管家。Windows（最常见）、macOS（苹果）、Linux（服务器/开发者）。所有操作都要经过它。

💡速记：硬盘存东西 → 打开程序 → 加载到内存 → CPU 执行 → 显示在屏幕。跟冰箱（硬盘）→ 拿菜到厨房台面（内存）→ 开火做菜（CPU）→ 装盘上桌（屏幕）一个道理。

1.2 文件系统 — 你电脑的「仓库」

计算机用文件夹（目录）组织一切。每个文件有唯一的路径。在编程里，路径写错是最常见的新手 bug。

系统路径示例分隔符
WindowsC:\Users\张三\Desktop\我的项目\\
macOS / Linux/home/zhangsan/projects//
📁

文件后缀名

.html=网页，.py=Python，.js=JavaScript，.cpp=C++，.json=数据。后缀名决定用哪个程序打开。

🔗

相对路径 vs 绝对路径

绝对路径：从根目录开始（C:\...）。相对路径：从当前目录开始（./data/file.json）。编程里一般用相对路径。

第 3 章

🌐 计算机网络 — 你的数据是怎么传出去的

前端调后端、浏览器发请求、服务器响应——这些背后全靠网络打交道。

3.1 HTTP — 互联网的通用语言

浏览器和服务器之间用 HTTP 通信。HTTPS=HTTP+加密（地址栏🔒）。HTTP 有 7 种常见方法（Method）：

方法作用对应数据库操作
GET读取数据SELECT
POST创建数据INSERT
PUT完整更新UPDATE(全字段)
PATCH部分更新UPDATE(部分字段)
DELETE删除数据DELETE

3.2 HTTP 状态码 — 一眼看出问题在哪

2xx 成功

200 OK — 正常
201 Created — 创建成功
204 No Content — 成功但无返回

3xx 重定向

301 永久搬家
302 临时搬家
304 用缓存

4xx 客户端错误

400 请求格式错
401 没认证
403 你没权限
404 找不到
429 请求太多被限

5xx 服务端错误

500 服务器崩了
502 网关坏了
503 服务器在维护
504 超时

3.3 DNS — 域名怎么变成 IP 的

你在浏览器打 google.com → DNS 服务器自动查出真正的 IP（如 142.250.72.46）→ 浏览器向这个 IP 发请求。DNS 像一个全球电话簿。

3.4 端口 — 一台服务器跑多个服务

IP 找到机器，端口找到这台机器上的具体服务：

端口服务
80HTTP（网页）
443HTTPS（加密网页）
22SSH（远程登录）
3306MySQL 数据库
3000/8000/8080开发常用端口
💡一句话记住网络：DNS 翻译域名→IP → HTTP 协议发请求 → 端口决定哪个服务处理 → 服务器处理完返回响应。整个过程通常不到 1 秒。
第 4 章

🔄 Git & GitHub — 代码的时光机

Git 是全球通用的版本控制系统。没有 Git，多人协作写代码是灾难。

4.1 Git 解决什么问题

📸

快照（Save Point）

每次 commit 就是给你的代码拍一张快照。改坏了随时回滚到任意历史版本。

🌿

分支（Branch）

从主线分出支线，在上面做实验。做好了合并（merge），做烂了删掉。互不干扰。

☁️

远程仓库（Remote）

代码存在 GitHub/Gitee 上。相当于代码的百度网盘+协作平台。

🤝

合并（Merge）

你的代码和他的代码自动合并。有冲突时 Git 会标出来，手动解决就行。

4.2 常用 Git 命令

终端 
git init              # 初始化（让当前目录变成仓库）
git add .             # 暂存所有改动
git commit -m "解决了bug"  # 提交（-m是备注）
git push              # 推到 GitHub
git pull              # 拉下别人的更新
git status            # 看当前状态
git log               # 看提交历史
git checkout -b xxx   # 创建并切换到新分支
git merge xxx         # 合并分支
git reset --hard HEAD~1  # 回到上一个版本（危险操作！）
⚠️千万不要把你的 API Key、密码、.env 文件提交到 GitHub 上。用 .gitignore 文件排除它们。已经提交的密钥=公开的秘密，必须立即删除+重新生成。

服务员（API）

5.3 JSON — API 对话的唯一语言

JSON

6.2 LLM 到底在干什么

LLM 是一个超强续写机器。你给它一段文字，它预测下一个最可能的词，一个词一个词续写下去。对代码、翻译、写作——全是同一个机制。

6.3 Token、Temperature、Context Window

🪙

Token（令牌）

模型的计价单位。1 token≈0.75个英文单词≈0.5个中文汉字。1000 token≈几厘钱（DeepSeek）。一句话总结：按字数收钱，但计价单位不是"字"而是"token"。

🌡️

Temperature（温度）

0~2之间。0=最保守（代码/事实），1=有创意的发散（写作/头脑风暴）。写代码设 0.0~0.3，写文章 0.7~1.0。

📐

Context Window

模型一次能"读到"的最大文字量。GPT-4o=128K(≈一本中篇小说)，Claude=200K，DeepSeek V4=1M(≈三体第一部)。context越大越贵。

⚙️

System Prompt

预设角色+规则。比如"你是资深前端工程师，用中文+emoji回复"。这不算在用户输入里，不计费。

第 14 章

🗄️ 数据库基础 — 你的数据存在哪

14.1 关系型 SQL vs 非关系型 NoSQL

📊

SQL（关系型）

数据像 Excel 表格，有严格的行列结构。MySQL、PostgreSQL、SQLite。适合有清晰结构的数据：用户表、订单表、商品表。用 SQL 语言查询。
SELECT * FROM users WHERE age>18

📦

NoSQL（非关系型）

数据像 JSON 文档，灵活随意。MongoDB、Redis、Firebase。适合结构不固定的数据：日志、用户行为、缓存。查询方式各不相同。
db.users.find({age:{$gt:18}})

14.2 新手应该用哪个

SQLite。不需要装服务器，一个文件就是整个数据库。适合个人项目、原型开发。Python 自带支持：import sqlite3。

Python — SQLite 示例 
import sqlite3
conn = sqlite3.connect("myapp.db")  # 文件即数据库
cur = conn.cursor()
cur.execute("CREATE TABLE users (id INTEGER PRIMARY KEY, name TEXT, age INTEGER)")
cur.execute("INSERT INTO users VALUES (1, '张三', 25)")
rows = cur.execute("SELECT * FROM users").fetchall()
print(rows)  # [(1, '张三', 25)]
conn.commit(); conn.close()
💡什么时候上真正的数据库？当你的项目需要多用户同时访问、数据量超过 10 万条、或者需要高可用时。一个人做的小项目 SQLite 完全够用。
第 15 章

🤖 AI Agent — 让 AI 自己干活

这是 AI 应用的最新形态。Agent 不是聊天——它能使用工具、制定计划、执行多步任务。

15.1 什么是 Agent

  ┌─────────────────────────────────────────────┐
  │            AI Agent 工作循环                 │
  │                                             │
  │  用户指令 → LLM思考 → 决定用哪个工具        │
  │      ↑                      ↓               │
  │      └── 返回结果 ← 执行工具（搜索/代码/API）│
  └─────────────────────────────────────────────┘

普通 LLM：我问你答，一问一答。
Agent：你给我目标，我自己查资料、写代码、调 API、修 bug、最终交付。中间不需要你插手。

15.2 Agent 的核心能力：工具调用（Function Calling）

Agent 不是说"我应该帮你搜一下"——它直接调搜索 API 搜了、把结果读出来了、基于结果继续干活。这就是 Tool Use（工具调用）：

Python — Agent 工具调用概念 
# 定义工具：Agent 能用的功能
tools = [{
    "type": "function",
    "function": {
        "name": "search_web",
        "description": "搜索互联网",
        "parameters": {"query": {"type": "string"}}
    }
}]
# LLM 决定调哪个工具 → 你执行工具 → 把结果返回给 LLM → 继续对话
# 循环直到 LLM 说 "任务完成"

15.3 多 Agent 系统

真正的 Agent 应用往往不止一个 Agent：

🏗️

架构师 Agent

分析需求 → 拆解任务 → 分派给子 Agent

💻

开发 Agent

写代码 → 运行测试 → 修 bug

🔍

审查 Agent

检查代码质量 → 发现安全问题 → 提出改进建议

📝

文档 Agent

自动生成 README → 写 API 文档

🤯这就是 Claude Code 里「子 Agent」的底层原理。你让它做一个网站，它内部可能调用一个前端 Agent 写 HTML，一个样式 Agent 写 CSS，一个逻辑 Agent 写 JS，然后合并给你。

15.4 MCP（Model Context Protocol）— Agent 的 USB 接口

MCP 是 Anthropic 推出的大模型统一工具接口标准。类比：USB 插口统一了键盘鼠标U盘的连接方式，MCP 统一了 AI 连接各种工具的方式。

✅MCP 的意义：不用每个 Agent 框架自己写一套工具调用。只要工具支持 MCP，任何支持 MCP 的 AI 都能直接用它。就像 USB 设备插任何电脑都能用。modelcontextprotocol.io
第 16 章

📚 RAG & 向量数据库 — 让 AI 读懂你的文档

16.1 RAG 是什么

RAG（Retrieval-Augmented Generation）= 检索增强生成。LLM 的知识只到训练日期截止。RAG 让 AI 能「现查现答」——先检索你的文档，再基于检索结果回答。

  用户问："公司今年的销售额是多少？"
     ↓
  ［RAG流程］
  1. 把问题转成向量（Embedding）
  2. 在向量库中搜索最相关的文档片段
  3. 找到 3 条相关记录 → 把这些记录+用户问题一起发给 LLM
  4. LLM 基于检索到的数据回答："今年销售额为 ¥520万，同比增长23%"
     ↓
  用户得到基于真实数据的准确答案

16.2 Embedding（嵌入）— 把文字变成数学向量

Embedding 是把文字转成一串数字，意义相近的文字在数学空间中距离更近。比如 "猫"和"猫咪"的向量很近，"猫"和"汽车"的向量很远。

Python — Embedding 示例 
from openai import OpenAI
client = OpenAI(api_key="sk-你的Key", base_url="https://api.deepseek.com/v1")
r = client.embeddings.create(model="text-embedding-3-small", input="你好世界")
print(len(r.data[0].embedding))  # 1536 个数字 = 这句话的"数学指纹"

16.3 向量数据库（Vector DB）— 专门存 Embedding 的数据库

数据库特点
Chroma轻量开源，适合学习和小项目
Pinecone托管服务，开箱即用，有免费额度
