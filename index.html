第 1 章

🌐 计算机网络 — 数据怎么从你到全世界

你点了"发送"，你的文字怎么到达地球另一边的服务器？这一章彻底讲清楚。

1.1 从一个聊天消息说起

你用微信给朋友发了一条"你好"。这条消息走了多远？

你的手机 → WiFi路由器 → 光猫 → 运营商基站 → 光纤 → 微信服务器 → 朋友手机
（30ms完成）   （0.1ms）         （5ms）      （10ms）   （15ms）

每一步，数据都以电磁波或光信号的形式在网线上跑。到了目的地再转回原来的文字。整个过程像快递——你的消息被打包、贴地址、运输、拆包、签收。

1.2 IP 地址 — 互联网的「门牌号」

每一台连接互联网的设备都有一个唯一的 IP 地址。就像你家有门牌号、你有手机号一样。没有 IP 地址，别的设备找不到你。

IPv4（老标准）用 4 个数字，每个 0-255。比如 Google 的 142.250.72.46。但全球只有约 43 亿个 IPv4 地址，不够用。所以有了 IPv6——用更长的数字串（如 2001:db8::1），地址多到地球上每一粒沙子都能分一个。

💡试试：打开终端输入 ping google.com，你就能看到 Google 的 IP 地址和到你电脑的延迟（ms）。数字越小越快。

1.3 DNS — 互联网的「电话簿」

你可能记得 google.com，但你记不住 142.250.72.46。DNS（域名系统）就是把你输入的网址翻译成 IP 地址的全球分布式电话簿。

你在浏览器打 "google.com"
         ↓
 ［路由器DNS缓存］有记录吗？→ 没有
         ↓
 ［运营商DNS服务器］有记录吗？→ 有！
         ↓
  返回 IP: 142.250.72.46
         ↓
  浏览器向这个 IP 发请求 → 页面显示

DNS 也能被劫持。你之前 WiFi 跳黄网——就是路由器 DNS 被改成了恶意服务器，把正常网址翻译成了色情网站的 IP。换成干净的 DNS（如 223.5.5.5 阿里、1.1.1.1 Cloudflare）就能解决。

1.4 HTTP — 浏览器和服务器之间的「快递协议」

IP 找到机器，但两台机器之间怎么对话？用 HTTP（超文本传输协议）。就像寄快递：你要填寄件人、收件人、物品描述。

一个真正的 HTTP 请求长这样：

HTTP 请求 
GET /index.html HTTP/1.1                        ← 方法 路径 版本
Host: www.example.com                            ← 找哪个网站
Accept: text/html                                ← 我要 HTML
User-Agent: Mozilla/5.0                          ← 我是浏览器

服务器收到后返回：

HTTP 响应 
HTTP/1.1 200 OK                                  ← 状态码 200=成功
Content-Type: text/html; charset=utf-8            ← 内容类型
Content-Length: 1234                              ← 内容大小

<html><body>...网页内容...</body></html>

1.5 HTTP 方法 — GET/POST/PUT/DELETE 到底什么意思

方法干啥的类比有请求体吗
GET读数据淘宝搜"鞋子"→看到列表❌ 没有
POST创建数据注册账号→填表→提交✅ 有
PUT完整更新把整篇文章换成新版本✅ 有
PATCH部分更新只改文章标题不改正文✅ 有
DELETE删除删帖子❌ 没有
💡GET 和 POST 的区别是最常考的面试题。GET 把参数放在网址后面（?name=张三&age=25），POST 把参数藏在请求体里。所以登录密码绝对不能用 GET——网址会被别人看到。用 POST。

1.6 HTTP 状态码 — 一眼看出问题在哪

2xx 成功

200 OK 正常
201 Created 创建成功
204 No Content 成功但没数据返回

3xx 重定向

301 永久搬家
302 临时搬家
304 用缓存别重发

4xx 你错了

400 请求格式错
401 没登录
403 没权限
404 页面不存在

5xx 服务器错了

500 服务器挂了
502 网关坏了
503 服务器维护中

记住这两组：404 是你找的页面不存在，500 是服务器自己出问题了。遇到 429 说明你请求太频繁被限速了，等一会再试。

1.7 端口 — 一台机器跑几十个服务

IP 找到哪台服务器，端口（Port）找到这台服务器上的哪个服务。

类比：IP=写字楼地址，端口=楼里的房间号。你寄快递要写「XX大楼 801 室」，发 HTTP 请求就是「IP 地址 + 端口 80」。

端口标准服务你见过吗
80HTTP（网页）每次打开网页
443HTTPS（加密网页）地址栏有🔒时
22SSH（远程登录服务器）程序员天天用
3000/8000/8080开发调试用你本地开发时

1.8 HTTPS 为什么比 HTTP 安全

HTTP 传输是明文的——任何人截获你的数据包都能直接看到内容。你在公共 WiFi 上输入的密码如果走 HTTP，隔壁桌的黑客拿个工具就能读。

HTTPS = HTTP + TLS 加密。数据在发送前加密，只有接收方有密钥能解开。中间截获的都是乱码。地址栏有🔒=HTTPS。现在几乎所有网站都是 HTTPS。

1.9 REST API — 当今最主流的 API 风格

REST = Representational State Transfer。听起来吓人，其实就几条规范：

REST API 设计示例 
# 获取所有用户  GET    /api/users
# 获取 ID=1 的用户GET    /api/users/1
# 创建用户       POST   /api/users           + 请求体（用户信息JSON）
# 更新用户       PUT    /api/users/1          + 请求体（完整新数据）
# 删除用户       DELETE /api/users/1

你调用 DeepSeek API 本质上就是：POST 一个 JSON 到 https://api.deepseek.com/v1/chat/completions → 服务器返回 JSON → 你的程序解析结果。

2.3 Git 到底解决什么问题

你写了一个网站，第二天改了 50 行代码，第三天发现改坏了——但已经不知道原来什么样了。Git 就是解决这个的。

每次你 git commit，Git 就给你的整个项目拍一张快照。什么时候改坏了，随时回到任意一张快照。你还可以创建分支（Branch）——从主线分出一条支线，在上面做实验。做好了合并回去，做烂了直接删分支，主线毫发无损。

GitHub 就是把你的快照存在云上。相当于百度网盘+协作平台。你把代码 push 到 GitHub，别人可以 pull 下来看、改、提合并请求。

2.4 Git 核心操作 — 你每天做的就这 4 步

⚠️千万不要用 git add . 就提交！提交前用 git status 看看哪些文件被改了。确保 .env（存 API Key 的文件）不在其中。API Key 上了 GitHub = 全世界都能看到。

2.6 GitHub Pages — 免费托管你的网站

只要你把 HTML 文件推到 GitHub，开启 GitHub Pages 功能，就能得到一个免费的公开网址：https://你的用户名.github.io/仓库名/。全世界都能打开，不用买服务器，不用备案。这是个人开发者最常用的免费部署方案。

  1. 创建 GitHub 仓库 → 设为 Public

3.5 函数 — 把常用逻辑封装成积木

def calculate_price(base, discount=0, tax=0.13):
    """计算最终价格。discount=折扣率，tax=税率"""
    discounted = base * (1 - discount)
    return discounted * (1 + tax)

result = calculate_price(100, discount=0.2)  # 100块打8折+13%税=90.4
print(result)  # 90.4

3.6 类（Class）— 把数据和操作打包

类=蓝图，实例=根据蓝图造出来的具体对象。"人"是一个类，"你"是这个类的实例。类定义了属性（name,age）和方法（speak()）。

Python 
class Student:
    def __init__(self, name, score):  # __init__ = 构造函数，创建实例时自动调用
        self.name = name; self.score = score
    def pass_or_not(self):  # 方法 = 类里定义的函数
        return "✅" if self.score >= 60 else "❌"
    def change_score(self, new_score):
        self.score = new_score  # 修改自己的属性

s1 = Student("张三", 85); s2 = Student("李四", 45)
print(s1.pass_or_not())  # ✅
print(s2.pass_or_not())  # ❌

3.8 Python 赚钱实操：网页爬虫

Python — 爬取商品信息 
import requests; from bs4 import BeautifulSoup; import csv
url = "https://books.toscrape.com"  # 练习用网站，随便爬不违法
resp = requests.get(url); soup = BeautifulSoup(resp.text, "html.parser")
books = []
for item in soup.select("article.product_pod"):
    books.append({"title":item.h3.a.attrs["title"], "price":item.select_one(".price_color").text})
with open("books.csv","w",newline="",encoding="utf-8") as f:
    w = csv.DictWriter(f, fieldnames=["title","price"]); w.writeheader(); w.writerows(books)
print(f"✅ 爬了{len(books)}本书")

注意：爬虫有法律风险。只爬公开数据，不爬需要登录的网站，不把数据转卖。遵守 robots.txt 协议。

第 4 章

🔌 API 彻底讲透 — 所有 AI 应用都建立在它上面

API = Application Programming Interface。别被名字唬住。它就是一个「服务员」。

4.1 为什么需要 API

你点外卖。你不进厨房、不开火、不买菜——你只要对着手机点两下，饭就送到门口。厨房和你之间有个中间层：菜单+外卖员。这个中间层就是 API。

同理，DeepSeek 的模型跑在云端的大 GPU 服务器上。你不可能在自己的笔记本上跑 6710 亿参数的模型。你只能发请求给 DeepSeek 的服务器，让它算完把结果返回给你。这个「发请求→收结果」的通道就是 API。

4.2 API Key — 不是订阅，是按量计费的预付费表

很多人以为 API Key 是"买了个会员"。不是的。API Key 像一张预付费电卡。你往账户里充值 ¥10，每次调用扣几分到几毛，余额归零就停。你不调用就不扣钱。

OpenAI 的 Key 以 sk- 开头，Anthropic 的 Key 以 sk-ant- 开头，DeepSeek 的也是 sk-。Key 是幂等凭证——拿到 Key = 能花钱 = 绝对不能泄露。

4.3 JSON — API 世界的通用语

你的程序说 Python 话，AI 服务器说 C++ 话，中间怎么沟通？用 JSON——一种极其简单的、纯文本的、人和机器都能读懂的格式。

JSON 只有 6 种类型：字符串（"hello"）、数字（42）、布尔（true/false）、数组（[1,2,3]）、对象（{"key":"value"}）、null（null）。一切都是键值对+嵌套。

JSON — 这就是 API 通信的格式 
{
  "messages": [
    { "role": "user", "content": "解释什么是API" }
  ],
  "model": "deepseek-chat",
  "temperature": 0.7,
  "stream": false
}

这个 JSON 翻译成人话："喂，DeepSeek，用 deepseek-chat 模型，温度 0.7，别流式，帮我回答这个问题：解释什么是 API。"——然后 DeepSeek 内部处理，返回另一个 JSON 给你，里面就有答案。

第 5 章

🧠 AI 理论 — 模型到底怎么「学」出来的

不要求你推导公式，但你要知道 Token 是什么、Context Window 怎么影响成本、Temperature 怎么调。

5.1 AI → 机器学习 → 深度学习 → LLM → Agent

这是一个嵌套关系。AI 是最大的圈，Agent 是最里面那个：

AI（人工智能）→ 让机器表现出智能，包罗万象：下棋、识图、翻译、开车...
  └── ML（机器学习）→ AI当前最成功的子领域：不靠人工写规则，让机器从数据里自动找规律
       └── DL（深度学习）→ ML 中最牛的子领域：用多层次的人工神经网络模拟人脑
            └── LLM（大语言模型）→ DL 的最新突破：用海量文本训练出的"写文章机器"
                 └── Agent（智能体）→ LLM + 工具调用：不只是聊天，能自主执行任务

5.2 LLM 的本质：超强「下一个词预测器」

不要神话大模型。它做的事情本质上就是：给定前文，猜下一个最可能的词。然后一个词一个词地续写下去，直到写完。它没有「意识」、不懂「意义」——但它见过万亿级别的文字，对词汇之间的概率关系掌握到极致。这种「概率续写」能力强大到让它看起来像在「思考」。

5.3 Token — 模型不是按字算钱的

用中文说："我喜欢你"→ 但模型内部不是按汉字处理的。它把文字切成 Token（子词级片段）。

大致换算：1 Token ≈ 0.75 个英文单词 ≈ 0.5 个中文字 ≈ 3-4 个英文字母。"Hello world" 约 3 Token，"你好世界" 约 4 Token。

你调一次 API，费用 =（输入 Token 数 + 输出 Token 数）× 单价。DeepSeek 的单价是每百万 Token 1-4 块钱。

💰成本实测：我问 DeepSeek "写一首关于编程的打油诗"，输入约 15 Token，输出约 120 Token，总消耗约 135 Token。DeepSeek V4 定价每百万 Token 约 ¥2，这次调用花了 135/1,000,000 × 2 ≈ ¥0.00027。不到 0.03 分钱。你充 ¥10，能这样聊几万次。

5.4 Temperature — 控制 AI 的「创意浓度」

Temperature 范围 0~2。原理：它影响模型选词的概率分布。

Temperature = 0 → 模型只选概率最高的词。输出稳定、可重复。适合：写代码、事实问答、翻译。
Temperature = 1 → 模型可能在"概率前几名"中随机选。有创意但如果选到低概率的词可能不准确。适合：写故事、诗歌、头脑风暴。
Temperature > 1 → 开始胡说八道。不建议。

🎯经验公式：写代码/数据提取/翻译 → 0.0~0.3。写文章/邮件/文案 → 0.7~0.9。头脑风暴/创意 → 1.0。

5.5 Context Window — 模型一次能「读」多少字

Context Window 越大，AI 越能理解你的整本书、整个项目的全部代码。但 Context 越大也越贵——因为你把更多 Token 放进去了。DeepSeek V4 的 1M Context 能一次性读完《三体》第一部（约 80 万字），Claude 200K 相当于一本中篇小说，GPT-4o 128K ≈ 300 页文档。

在实际开发中：Context Window = 你的整个项目代码能否一次性喂给 AI。如果项目超过 Context Window，你需要做 RAG（后面讲）。

7.3 多轮对话 — 让 AI 记住上下文

AI 本身无状态——每次请求它是"失忆"的。要让 AI "记住"上下文，你需要把前面的对话也放进 messages 数组里。

Python 
history = ["user:你好", "assistant:你好！有什么可以帮你？"]
msgs = [{"role":"system", "content": "你是日语老师"}]
for line in history:
    role, content = line.split(":", 1)
    if role == "user": msgs.append({"role":"user", "content": content})
    else: msgs.append({"role":"assistant", "content": content})
msgs.append({"role":"user", "content": "刚教的3个单词是什么？"})
r = client.chat.completions.create(model="deepseek-chat", messages=msgs)

7.4 流式输出 & Function Calling

流式（Streaming）：打开一个开关，AI 就逐字返回结果而非一次性全部返回。用户体验像打字机——ChatGPT 就是这样的。

Python — 流式输出 
stream = client.chat.completions.create(model="deepseek-chat",
    messages=[{"role":"user", "content": "写一首编程打油诗"}], stream=True)
for chunk in stream:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="", flush=True)

Function Calling（工具调用）：告诉 AI "你有这些工具可用"，AI 决定调用哪个工具、传什么参数，你的代码执行后把结果返回给 AI 继续对话。这是 Agent 的核心机制（第13章详讲）。

第 11 章

🗄️ 数据库 — 数据存哪、怎么查

11.1 SQL vs NoSQL — 什么时候用哪个

📊

SQL（关系型）

数据像 Excel 表格，有固定的列。数据之间有关联（外键）。MySQL、PostgreSQL、SQLite。适合：用户表、订单表、任何结构固定的数据。

📦

NoSQL（非关系型）

数据像 JSON 文档。结构不固定，随时能加字段。MongoDB、Redis、Firebase。适合：日志、用户行为、缓存、实时数据。

11.2 SQLite — 新手之神，一行代码建数据库

SQLite 不需要装服务器——一个文件就是整个数据库。Python 自带 import sqlite3。个人项目、原型、小工具，SQLite 完全够用。

Python + SQLite 
import sqlite3
conn = sqlite3.connect("myapp.db")  # 文件就是数据库！没有MySQL服务器
c = conn.cursor()
c.execute("CREATE TABLE IF NOT EXISTS users (id INTEGER PRIMARY KEY, name TEXT, age INTEGER)")
c.execute("INSERT INTO users VALUES (1, '张三', 25), (2, '李四', 30)")
rows = c.execute("SELECT * FROM users WHERE age > 20").fetchall()
for row in rows: print(row)  # (1, '张三', 25)  (2, '李四', 30)
conn.commit(); conn.close()

11.3 什么时候该升级到 PostgreSQL / MongoDB

当你的项目需要多用户同时读写（并发）、数据量超过 10 万条、需要高可用（服务器挂了自动切到备用）、或者有复杂的关联查询时。一个人的项目 SQLite 完全够。

第 13 章

🤖 AI Agent — 让 AI 自主干活

13.1 Agent 是什么

普通 LLM：你问→它答→你问→它答……每一轮都靠你驱动。
Agent：你给一个目标→它自己规划步骤→自己调用工具→自己判断结果是否正确→循环直到目标达成。你在中途不需要插手。

用户: "帮我做市场调研：上周AI行业有什么大新闻？整理成报告发邮件给我。"
  ↓
Agent: ① 调用 web_search 工具搜索 "AI news last week"
     ② 读搜索结果，提取关键信息
     ③ 调用 write_file 工具写报告
     ④ 调用 send_email 工具发送
     ⑤ 报告用户 "✅ 已完成，请查收邮件"

13.2 Agent 的核心：ReAct 循环

Agent 最常见的决策框架叫 ReAct（Reasoning + Acting）：

  1. 思考（Thought）：我现在该做什么？有哪些可用工具？
  2. 行动（Action）：调用一个工具（搜索API、写文件、运行代码…）
  3. 观察（Observation）：工具返回了什么结果？
  4. 回到步骤 1，直到目标达成。

13.3 MCP（Model Context Protocol）

MCP 是 Anthropic 推出的模型→工具的连接标准。类比：USB 统一了电脑和键盘/鼠标/U盘的连接方式，MCP 统一了 AI 和各种外部工具的连接方式。只要一个工具支持 MCP，任何支持 MCP 的 AI 程序都能直接调用它——不需要为每个 AI 重写适配代码。

📍官方文档：modelcontextprotocol.io。Claude Code、Cursor、OpenClaw 都已经支持 MCP。
第 14 章

📚 RAG — 让 AI 读懂你的文档

14.1 为什么需要 RAG

LLM 的知识只到训练截止日——它不知道你公司的内部销售数据，不知道你的产品文档。RAG 在 LLM 收到问题后先检索相关文档，把检索到的文档拼到 prompt 里，让 LLM 基于文档回答。答案不再是"编"的——是有据可查的。

14.2 Embedding — 把文字变成数学向量

Embedding 是把任意文本转成一串数字（向量），语义相近的文本在向量空间中距离更近。比如："猫"和"猫咪"的向量几乎重叠，"猫"和"汽车"的向量差很远。

Python — 生成 Embedding 
from openai import OpenAI
client = OpenAI(api_key="sk-", base_url="https://api.deepseek.com/v1")
r = client.embeddings.create(model="text-embedding-3-small", input="你好世界")
print(len(r.data[0].embedding))  # 1536个数字 = 这句话的语义指纹

14.3 RAG 完整流程

  1. 分块（Chunking）：把你所有的文档切成小段（每段 500-1000 字）
  2. 向量化（Embedding）：每一段转成向量，存进向量数据库（Chroma/Pinecone）
  3. 检索（Retrieval）：用户提问→把问题也转成向量→在数据库里找最相似的 N 个文档片段
  4. 生成（Generation）：把「检索结果 + 用户问题」一起发给 LLM→它基于文档回答

14.4 向量数据库选型

数据库适合部署
Chroma学习/小项目Python 一行 pip install chromadb
Pinecone生产环境托管服务，有免费额度
Qdrant高性能自托管或云服务
Supabase pgvector已用 PostgreSQL直接扩展，不用新数据库
